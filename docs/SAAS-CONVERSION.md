# SaaS conversion assessment — multi-tenant Marketing Command Centre

**Question:** what is the effort to convert the single-group platform into a SaaS
for (a) marketing agencies managing many clients and (b) owners of multiple
businesses?

**Answer up front:** roughly **6–8 build phases of the size we've been shipping**
— i.e. ~55–65% of the effort invested in Phases 1–12 — plus one external
calendar dependency (Meta App Review, 2–6 weeks, runs in parallel). It is a
scoping-layer + commercial-shell conversion, **not a rewrite**: the whole value
core (content pipeline, compliance engine, approval routing, publishing engine,
asset usage-rights, automation) is already company-scoped and tenant-neutral.

Grounding: a 5-agent code audit (2026-07-04) inventoried **138 concrete
single-tenant assumptions** — 5 trivial, 76 small, 42 medium, 15 large. The
raw inventory matters less than the shape: most "small" findings collapse into
three systematic changes (repo scoping, RLS rewrite, singleton conversion).

---

## The tenancy model (recommended design)

Add ONE new entity — `tenants` (a.k.a. organisation/workspace) — and derive
almost everything through the existing company scoping:

```
tenants (id, name, plan, status, branding…)
  └── companies.tenant_id  (NOT NULL FK)      ← the one structural column
  └── tenant_members (user_id, tenant_id, tenant_role)   ← replaces "role means whole deployment"
```

**Key insight from the audit:** every content-bearing entity already carries
`companyId`. Tenancy therefore derives via `company → tenant` for ~20 tables.
`tenant_id` columns are only *required* where company linkage is absent or
nullable:

| Needs explicit `tenant_id` | Why |
|---|---|
| `companies` | the structural link |
| `tenant_members` (new) | user↔tenant with per-tenant role |
| `publishing_controls`, `security_settings`, `automation_settings` | today single-row singletons → one row per tenant |
| `approved_responses` / `brand_templates` / `prompt_templates` with `company_id = null` | "group-wide" becomes "tenant-wide" |
| `audit_logs`, `ai_runs` (null-company rows), `automation_runs`, `legal_holds` | must never leak across tenants |

RLS: `has_company_access(cid)` gains a tenant check; `is_admin()` becomes
`is_tenant_admin(tid)`; a new `platform_admin` (us, the operator) sits above.
Recommended isolation strategy: **single database, tenant_id + RLS** — Supabase-
native, standard at this scale; schema-per-tenant is not warranted.

### Role model change
| Today | SaaS |
|---|---|
| `super_admin` (deployment-wide) | **platform_admin** (operator; support/ops only) |
| `admin` (whole group) | **tenant admin / owner** (their tenant only) |
| senior/compliance approvals → super_admin | → tenant owner (agencies WANT approvals inside the agency) |
| `user` scoped by `company_access` | unchanged, plus tenant membership |
| email globally unique | unique **per tenant** (an agency user and a client user may share an email) |
